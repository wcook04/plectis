What you can actually check

A way in, and a way to check it.

Plectis comes from a larger AI-native system I have been building with AI. I have not put all of it online: it includes live tools and non-public data, and releasing the whole thing at once did not seem sensible or responsible. This is a smaller public executable cross-section instead, built around source-linked components, public-safe stubs, and small synthetic fixtures, so the mechanisms can be inspected without publishing the live working environment.

What can I run?

The fastest way to understand Plectis is not to read all 88 component records. Clone the repository, run the quickstart check, then follow one mechanism to its evidence line, source path, and scope limit.

  1. 1Clone the repository.
  2. 2Run the quickstart check and inspect the result record.
  3. 3Read the evidence kind, rank, real-tool marker, and the line that says what the pass does not prove.
  4. 4Inspect one mechanism card, such as the Lean proof-search lab runtime, and follow its source link.
  5. 5Read where the public record stops, and check the site never claims past it.

Some components run real tools: the Lean prover and Lake, finance statistics code, git provenance checks, or copied modules run in process. Others derive a verdict over copied source or a declared contract against small synthetic fixtures. Each card says which, in plain language. The populated slice runs locally against a folder and writes readable state without external model calls.

What evidence does a record carry?

Each component declares its own evidence: a class, a strength rank from 1 to 5 for how independently it is checked, and a line on what it does not prove. The card gives you that, so you can judge it on the evidence rather than the wording.

How to read the rank. Rank measures verdict independence, not how much machinery a component runs or how mature it is: how far a check can reach its own pass or fail rather than echo an answer a fixture handed it, from 1, where the fixture supplies the answer, to 5, where the check derives the whole verdict with nothing fed to it. A component can run real tools and still sit at a bounded rank; a validator with no tool can rank higher when it checks more of its own contract unaided.

Where does the public version stop?

The fuller system is not all here, and that is on purpose. The fixtures are there for a reason: they are how a piece of the work can be shown without shipping the live system it normally runs against. Where a component only checks the shape of a fixture, its card says so. A fixture-bound replay is the public-safe form of a mechanism that should not be disclosed as live private-system authority: it exposes the mechanism, the boundary, and the refusal, not a running service.

One place to see that line held is the Agent Claim Audit Casebook, released alongside the site: redacted, replayable cases where an agent completion note or a monitor sign-off sounds finished, and the evidence records say something smaller. One real trace whose "committed, clean" note lacked a final delta and any terminal validation is lowered, not accepted; another plants fabricated commit, ledger, and test-pass claims that real git and pytest checks block with typed error codes. It does not claim general agent honesty or deception detection; each case states what it does not prove.

Where the site makes a claim, it tries to give you enough of the source trail to check it yourself, including the point where the public version stops. If something here is confusing, or claims more than it shows, I would rather hear it.

Questions worth asking it

Four sets for four reasons to be here; any question from any set is fair to ask cold.

  • What is this, in plain terms?
  • Where should I start?
  • Walk me through one component, end to end.

With a coding agent in the repository

The sets above are questions for an assistant reading the map. A coding agent (Claude Code, Codex, Cursor) works inside a repository itself: one prompt sets it up, and after that you assign work rather than asking about a file. How that works.

Clone https://github.com/wcook04/plectis and open it. Read AGENTS.md and README.md to orient yourself, run the quickstart check, and show me what the result record says. Then take my questions.

Once a populated source tree is in front of it, work worth assigning, one item at a time:

  • Run one component end to end and walk me through the evidence it leaves.
  • Take one claim from the website and verify it against the source here.
  • Which validators can you run right now, and what would each one prove?
  • Make one check fail on purpose and show me how it refuses.

Examples, not a script: ask anything, and ask follow-ups. The packet routes each question to evidence; the verdict stays the assistant's.

Read it sceptically, in both directions: a rank can overstate a component, but the plain wording can also undersell one. The system is built to be held to that standard, so it seems fair to do the same. Or hand it to your own assistant and ask it to argue against me:

Read this repository as a calibration reviewer. First, by area, name the strongest components and what each actually does, with its evidence class. Then judge both directions: where the wording claims more than the evidence shows, and where it undersells what a component demonstrates. Do not reduce the system to a set of projections; computed projection is one evidence class among several.
