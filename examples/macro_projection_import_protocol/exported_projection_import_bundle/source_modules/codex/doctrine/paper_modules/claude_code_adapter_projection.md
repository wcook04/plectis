## TLDR (compressed view)

`CLAUDE.md` is the **Claude Code actor adapter** over the shared hub: it carries Claude-specific deltas only and projects three managed regions from `codex/doctrine/agent_bootstrap.json` via the bootstrap builder — `claude_adapter_live` (live phase + actor context + bootstrap sequence), `paper_module_index` (compact discoverability slice), and `generated_routing` (routing rows mirrored from AGENTS.md). The hand-authored layers above and below the live block are the Claude-only additions: a preamble carrying the Rosetta-Stone Prime Directives table, follow-on register, and substrate-specific entry-affordance rows; and Claude-specific deltas covering hook semantics (`.claude/hooks/runtime_hook.py`), dispatch-yield-resume (`docs/bridge_resume_protocol.md`), and the resume control plane (`tools/meta/bridge/bridge_resume.py`). The schema hinge (actor contexts, MRS ids, bootstrap sequence, projection-vs-delta classification) lives in `codex_agent_bootstrap_surface_schema`; this module owns the Claude-side shape only — what the live block must carry, which sections are hand-editable, and which paths are Claude-only continuity surfaces.

## Intent

`CLAUDE.md` exists because Claude Code needs an actor-scoped adapter that reads `AGENTS.md` for shared doctrine but expresses Claude-specific hook wiring, continuity protocol, and resume control plane without polluting the shared hub. Three failure modes motivate the projection contract: (1) hand-edits to live regions get silently overwritten on the next builder run; (2) a Claude-only delta (hook entrypoint, transport adapter) leaking into AGENTS.md confuses Codex; (3) drift between the schema config and what `CLAUDE.md` actually shows leaves cold agents trusting stale facts. This module projects the schema hinge into Claude's specific shape so authors editing `CLAUDE.md` know which sections are theirs and which the builder owns.

### Invariants

- **Live blocks are projections — never hand-edit.** Any text inside `<!-- BEGIN claude_adapter_live --> … <!-- END claude_adapter_live -->` or the paper_module_index block is overwritten on every builder run. Editing the source of truth means editing `agent_bootstrap.json` (for live block) or authoring/refreshing paper modules (for paper_module_index).
- **Claude deltas stay Claude-only.** Hook semantics, transport-first rehydration, the resume control plane, and the `.claude/follow_on/` register are Claude-specific. They do not belong in `AGENTS.md` (shared) or `CODEX.md` (Codex actor).
- **Adapter `Depends on:` the schema hinge.** This module reads `codex_agent_bootstrap_surface_schema` for actor-context table, MRS ids, bootstrap sequence, and projection-vs-delta rules. If those drift, refresh the hinge and rerun the builder.
- **Claude reads `AGENTS.md` after the live block.** The mandatory read order in the preamble points cold Claude agents to the shared hub on step 3 — Claude is not a self-sufficient actor, it is an actor that augments the shared world model with deltas.
- **Continuity surfaces are pre-authored, not negotiated.** The Claude-specific continuity surfaces table (resume control plane, transport adapter, session transport artifact, hook heartbeat) is hand-authored doctrine; the builder does not generate it. If the surfaces move, edit `CLAUDE.md` Claude-deltas section by hand.

## Gap (what Will is signaling)

Will's signal here is the same as the parent index: cold agents must land on a self-describing entry surface without probing. The Claude-specific delta is that hook wiring, transport-first rehydration, and the disk-first re-entry contract are not portable across actors — they have to live in `CLAUDE.md` deltas, not in `AGENTS.md` shared doctrine. The deeper gap is that the Claude-deltas section still grows organically; a future refinement is to convert the substrate-specific entry-affordance table in the preamble into a builder-projected slice so that adding a new Claude-only subsystem also auto-projects into the adapter rather than requiring a hand edit. Anchor: `dtx_011` in `documentation_theory_index.json`, par `par_phase_08_raw_seed__source_1_next_ideas_md_020`.

## What a cold agent should NOT re-derive

- That `CLAUDE.md` carries Claude-specific deltas only — `AGENTS.md` is the shared hub, `CODEX.md` is the Codex adapter, and `AGENTS.override.md` is the Codex compact discovery seed.
- That live blocks are builder-owned — hand edits to `claude_adapter_live` or `paper_module_index` regions are silently overwritten.
- Where the Claude resume control plane lives — `tools/meta/bridge/bridge_resume.py` is the canonical entry, with status / jobs / clear-target subcommands.
- Whether to read `AGENTS.md` — yes, on step 3 of the mandatory read order; Claude is an adapter actor, not a self-sufficient one.
- That the hook entrypoint is `.claude/hooks/runtime_hook.py`, single dispatcher across 8 events, stdlib-only, bounded rehydration.

## Refresh contract

Refresh triggers:
- `agent_bootstrap.json::actor_context_surfaces` for `claude_code` changes — adapter live block needs rerun of the builder.
- A new Claude-only continuity surface is added — update the Claude-deltas continuity table in `CLAUDE.md` by hand.
- The hook contract changes (new event, new constant, new transport import) — update the Claude deltas section by hand.
- A new paper module is authored that should appear in the `paper_module_index` slice — author the module, then run `./repo-python tools/meta/factory/build_paper_module_index.py && ./repo-python tools/meta/factory/build_agent_bootstrap_projection.py`.
- The schema hinge in `codex_agent_bootstrap_surface_schema` changes — refresh that hinge first, then this module.

Stale signals:
- A row inside `<!-- BEGIN claude_adapter_live --> … <!-- END claude_adapter_live -->` does not match `agent_bootstrap.json::actor_context_surfaces` after a builder run.
- A path in the Claude-deltas continuity surfaces table 404s.
- The hook event count or constant in the deltas section drifts from `.claude/hooks/runtime_hook.py`.
